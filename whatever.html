Primitives

One recipe, five screens

The HUD, the build bar, the tower card, the Command Core panel, the profile panel, the About/Analytics panel, the win/lose banner, the settings panel, the avatar menu, the confirm modal — ten different surfaces in css/style.css, all built from one glass recipe and one accent color instead of seven bespoke looks. This page is built from the same recipe — the cards below are the actual CSS values, not a mockup.

Shared glass recipe
blur(18px)
backdrop-filter blur(18px) · the same two-stop dark gradient · 1px rgba(255,255,255,.06) border · the same inset+drop shadow · 18px radius. One selector list in style.css draws all ten surfaces — and every card on this page.

One accent color

#7ED9FF
HUD stat values · the in-game HUD's DOCS/ABOUT links · tower/room panel titles · every hover border on a build/tech/skill button · the achievement wall's "special" tier · the restart button · the About panel's category bars (four opacity steps of this one hex, not four new hues) — and this page's own accent, replacing an earlier draft that used an unrelated pink. A dozen-plus call sites, one hex value.

One button-state pattern

IdleHover DisabledMaxed
Hover: border + tint to the accent. Disabled: opacity .35–.4, cursor not-allowed. Maxed: green border, no further spend. Every interactive button in the Core screen, the profile panel and the build bar reads this way.
Found while building the previous version of this page: .profile-panel (added in Phase 4) had drifted from the pattern — it duplicated the shared glass recipe line-for-line instead of joining the shared selector like every other surface. Folded it in and deleted the duplicate block. Then this page itself drifted from the pattern the same way — it invented its own pink/rounded-font palette instead of the game's real one. Re-themed here to match. That's what this section is actually for: not just claiming consistency, but somewhere to catch it when it slips, including in the doc page auditing it.
How it's built

Real systems, not a roadmap

These pages aren't drawn in a diagramming tool — they're data, rendered once by a single, dumb renderer that draws any flow.

Primitive 1

Docs are data

Every flow is a plain object in this page's FLOWS array — lanes, nodes, edges, a code reference per step. Adding the next system's flow is adding data, not drawing a new diagram by hand.

Primitive 2

One dumb renderer

The same renderer — hover spotlight, curved edges, the sticky inspector — draws every flow on this page from the same JSON shape. No charting library, nothing to keep in sync by hand when a system changes.

Primitive 3

Real systems only

Nothing in WhatHow is aspirational. Every flow covers code that has actually shipped and is exercised by the test suite below — in-progress and future work stays on the Roadmap above, clearly marked as such, not blended in.
