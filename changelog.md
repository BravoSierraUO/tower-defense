- v0.9 - **Phase 3: Full Core + Room Variety.** 5 new Command Core rooms (Lab, Factory, Hangar, Shield, Dock), each 3-tier like the originals. Lab produces `researchRate`, accruing into a spendable `CommandCore.research` stockpile (`commandCore.update(dt)`, mirrors how `world.gold` works). A 5-node tech tree (`CONFIG.TECH_TREE`) gates Factory → {Hangar, Shield} → Dock behind research spend, plus a parallel Module Slots node; Lab itself needs no node since it's the free root (no Lab built = no researchRate = tree never advances, gate enough on its own). Factory's `buildTimeReduction` shortens every future room's construction timer; rooms now take real time to finish and contribute nothing to `totals()` until built. Shield's `shieldPct` cuts base damage in `resolveBaseHits`; Hangar's `dronePower` is passive DPS applied every frame to whichever live enemy is closest to the base (no projectile/visual — abstract drones). Dock adds a manual gold→research trade button, ratio scaling with its tier. Module slots: tier-gated capacity (0/1/2), one universal +20%-output module per slot, gold cost scaling with tier & existing module count. Room building/upgrading/modding is now a real gold sink for **all** rooms (including the original 3 starters) — previously free. - `config.js`, `room.js`, `commandcore.js`, `world.js`, `combat.js`, `game.js`, `input.js`, `renderer.js`, `ui.js`, `index.html`, `css/style.css`, `docs.html`
- v0.9.1 - **Test harness (Phase 5 item, pulled forward).** Node is now installed on the dev machine, and Node 20 ships a built-in test runner — no npm/deps needed. 52 tests: `commandcore.test.mjs` (room placement/tiers/build timers, tech-tree gating incl. a cycle check, module slots), `economy.test.mjs` (tower + room/upgrade/module/dock-trade gold gating, Command Core → economy wiring), `combat.test.mjs` (tower targeting/cooldown, Projectile damage-on-arrival, base-hit `hasHitBase` guard, Shield/Hangar effects), `spawner.test.mjs` (wave state machine, difficulty-tier unlock gating). `balance.test.mjs` is the one this was really for: regression guards that catch the economy silently breaking as constants get tuned — `towerCost()`/Shield's damage reduction can't hit a 0 or ≥100% floor even at a maxed-out Core, purchase costs (`buildCost`/`upgradeCost`/`moduleCost`) must never get cheaper the more you buy, and the full tech tree must stay reachable in a bounded amount of Lab accrual time. Run via `node --test tests/` (`npm test` also works — added a minimal `package.json` for the script alias, but npm itself isn't on this machine, so `node --test` is the primary path). `docs.html`'s Phase 5 card marks the test-harness item done. - `tests/helpers.mjs` (new), `tests/commandcore.test.mjs` (new), `tests/economy.test.mjs` (new), `tests/combat.test.mjs` (new), `tests/spawner.test.mjs` (new), `tests/balance.test.mjs` (new), `package.json` (new), `docs.html`
- v1.0 - **Phase 4: Progression — Levels, Achievements & Prestige.** Persistent player profile (`localStorage`, survives run restarts), skeleton ported from a sister project's (RMUV) leveling/achievement/prestige system — stripped of everything that was really about RMUV's 40-game arcade catalog (avatars, cosmetics, genre shelves, cross-game score normalization); this game has one CP source, so it needed none of that. A single Cascade-Point spine (`profile.js`) fed by run events (kills, wave clears, towers placed, rooms built, tech unlocked, run win/loss) drives a gentle sqrt level curve. A data-driven achievement registry (`achievements.js`, 21 badges across common/special/epic/legendary) is tested via a fixpoint loop on every event so one unlock can cascade into another the same tick. Prestige resets the CP climb at an escalating level gate (10, 20, 30…) for a permanent +10%/prestige CP-gain bonus plus spendable skill points, sunk into a 4-node skill tree (Gold/Damage/Build/Fortification Mastery) that stacks with the existing Command Core bonuses. Runs can now actually be replayed: added a "Play Again" button (previously the win/lose banner was a dead end requiring a page reload) that rebuilds World/CommandCore while keeping the profile — this is also what prestige uses internally. New `P` key opens the profile view (level/XP, prestige, skill tree, achievement grid) alongside the existing `B` (Command Core) toggle. The engine files stay decoupled from the profile the same way `ui.js` already stays decoupled from gameplay mutation: `game.js` diff-watches a handful of monotonic counters (`world.kills`, `world.towersPlaced`, `spawner.wavesCleared`, `commandCore.rooms.length`, `commandCore.unlockedTech.size`) each frame and translates deltas into `profile.emit()` calls, rather than threading Profile through combat/spawner internals. 21 new tests (`profile.test.mjs`) cover the level curve, CP economy, achievement fixpoint unlocking, prestige reset/payout/gate escalation, skill cost curve, and the skill bonuses actually reaching World/combat formulas. - `profile.js` (new), `achievements.js` (new), `config.js`, `world.js`, `spawner.js`, `combat.js`, `game.js`, `ui.js`, `index.html`, `css/style.css`, `tests/profile.test.mjs` (new), `tests/helpers.mjs`, `docs.html`

## Notes for next step

- **Phase 4 scope cut, on purpose:** RMUV's prestige also wipes per-game play stats so achievements re-earn across "lifetimes" (mastery stars, gold/silver/bronze per re-earn). That's real design value but real added complexity — a parallel life/permanent stat split. Skipped here: `profile.js`'s `life` stats and earned badges are permanent, prestige only resets `cp`. Revisit if prestiging more than 2-3 times ends up feeling like "nothing changed except a number."
- `profile.js`'s Node-safe storage fallback matters more than it looks: Node has no global `localStorage`, so `Profile` falls back to a per-instance in-memory store (NOT a shared module singleton — an earlier draft got this wrong and leaked state between test files) when `typeof localStorage === 'undefined'`. `World`/`freshGame()` in `tests/helpers.mjs` both default to a fresh `Profile` so every pre-existing test kept working unmodified.
- Couldn't screenshot-verify the profile panel/toast/restart-button UI in this environment — same sandboxed Chrome `ld.so` relocation crash noted since v0.8, unrelated to this change. Verified instead via the Node harness (`profile.js`/`achievements.js` have zero DOM dependencies) plus a manual cross-check that every `ui.js` `getElementById` call resolves against `index.html`.
- `spawner.js` picks difficulty tiers by weighted random from `CONFIG.DIFFICULTY_TIERS`, gated by `unlockWave`. This is deliberately simple — replace with real curve design once balance passes start.
- Base is currently single, at world center (0,0). Multiplayer/multi-base is out of scope until the single-base loop is fully working.
- Command Core still only allows one of each room type (`CommandCore.isBuilt`), now across 8 types instead of 3. Deliberately kept for Phase 3 rather than switching to build-many-of-each — tier upgrades + module slots already give each room type a progression curve, and one-of-each keeps the 8x8 grid from becoming a stacking optimization problem. Revisit if a future phase wants base-building depth beyond that.
- Couldn't screenshot-verify the new rooms/tech-tree/module UI in this environment — the sandboxed Chrome binary still crashes on launch (`ld.so` relocation failure), same as noted in v0.8, unrelated to this change. Verified instead with a Node harness (`js/config.js`, `room.js`, `commandcore.js`, `world.js`, `combat.js` have zero DOM dependencies, so they import and run directly under plain Node) that built rooms, ticked build timers and research accrual, unlocked tech nodes in dependency order, installed a module and checked its output bonus, and asserted Shield's damage reduction and Hangar's drone DPS against the actual numbers combat.js produced — plus a manual ID cross-check between `ui.js`'s `getElementById` calls and `index.html`. The DOM-dependent wiring (`ui.js` event listeners, `renderer.js` canvas layout, `game.js` input routing) was verified by code trace only, not a live click-through.
- Room build/upgrade/module costs and the build timer apply uniformly to all 8 room types, including the original Phase 2a starters (Reactor/AI Core/Storage), which were previously free and instant. This was a deliberate call to keep the Core's economy internally coherent now that it's a real gold sink — not called out as a balance concern since nothing has been playtested yet anyway (see the difficulty-curve note above).
- Shield and Hangar are combat-side modifiers (read via `world.commandCore.totals()` directly in `combat.js`), not routed through `world.js`'s gold-focused economy helpers — they don't touch gold, so they didn't need a `world.js` wrapper the way tower/room costs do.
- `docs.html`'s Towers/Base canvas flow diagrams (the node-graph tabs, not the Economy web) still don't show Shield's damage-reduction or Hangar's drone-DPS steps — only the Economy tab and Roadmap were updated. Low priority since those diagrams are about the tower/base combat path specifically, and Shield/Hangar hook in as flat modifiers rather than new steps in that flow, but worth a pass later if the diagrams start feeling stale.
- `World` now takes `commandCore` in its constructor (`new World(commandCore)`) so it can read `commandCore.totals()` for cost/reward/cap formulas — if `World` is ever instantiated elsewhere (tests, etc.), it needs a `CommandCore` instance passed in.
- Tower sell is right-click on a placed tower (60% refund of what was actually paid, tracked per-tower as `tower.cost`) — no confirmation dialog, matches the no-friction feel of placement.
- The test suite's own first draft had 2 bugs, both in test code, not game code: `balance.test.mjs`'s "max out a room" helper tried to build Shield/Factory/Hangar without unlocking their tech first (those room types return `null` from `placeRoom` when locked, same as the real game would refuse); fixed by unlocking the prereq chain before placing. `commandcore.test.mjs`'s cycle-detection check used a single global "seen" set across the whole tree walk, which flags any node reached twice as a "cycle" — but `dockAccess` legitimately reaches `factoryAccess` via two different branches (`hangarAccess` and `shieldAccess`), a valid diamond dependency, not a cycle. Fixed by tracking only the current DFS path per branch instead of a global visited set. Worth remembering if the tech tree grows and a real cycle check is needed elsewhere.
